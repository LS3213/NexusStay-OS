# About NexusStay OS

NexusStay OS 是一个面向小型酒店、旅馆、民宿的前台管理系统。项目从一个简单的 JSON 文件 + 单页 HTML 起步，经过多轮迭代，逐步演化为一个功能完整、可部署于生产环境的酒店管理平台，并提供三种技术栈版本以适配不同的部署场景。

---

## 项目起源

项目的起点是一个纯前端的房间分配工具——所有数据存在浏览器 localStorage 中，刷新即丢，没有用户系统，没有后台管理。目标是将其改造为一个真正的后台管理系统：有服务端、有数据库、有权限控制、有完整的业务流程。

---

## 核心设计理念

### 1. 零框架依赖

整个项目不使用任何后端框架（Express、Koa、Laravel 等），也不使用前端构建工具（Webpack、Vite 等）。后端使用语言原生 HTTP 能力，前端通过 CDN 引入 React 和 Tailwind CSS，浏览器端编译 JSX。这样做的好处是：

- 部署极其简单，无需 `npm run build`，无需配置构建管线
- 代码量可控，全部业务逻辑集中在一个文件中
- 适合小型项目，不需要理解框架约定

### 2. 单文件后端

无论是 Node.js 版的 `server.js` 还是 PHP 版的 `api.php`，所有后端逻辑（路由、认证、业务、数据库、WebSocket/SSE）都在一个文件中完成。这是刻意的设计选择——对于 80 间房的小旅馆来说，微服务架构是过度工程化。

### 3. 前后端分离但同域部署

前端是纯静态 HTML 文件，通过 `fetch` 调用后端 API。两者部署在同一个域名下，不存在跨域问题。前端不依赖后端的模板渲染引擎，后端只提供 JSON API。

### 4. 渐进式功能演化

项目经历了以下演化路径：

```
纯前端 localStorage → Node.js + JSON 文件 → Node.js + MySQL → PHP + MySQL
     (无后端)           (有后端无数据库)       (生产级)         (宝塔友好)
```

每个阶段都是对前一阶段的完整替代，不是增量补丁。功能只增不减，UI 逐步优化。

### 认证流程

```
客户端                        服务端
  │                              │
  │  POST /api/auth/login        │
  │  {username, password}        │
  │─────────────────────────────→│
  │                              │  验证密码哈希
  │                              │  生成随机 token
  │                              │  存入 sessions 表/Map
  │  Set-Cookie: ns_session=xxx  │
  │←─────────────────────────────│
  │                              │
  │  GET /api/inventory          │
  │  Cookie: ns_session=xxx      │
  │─────────────────────────────→│
  │                              │  从 cookie 取 token
  │                              │  查 sessions 表/Map
  │                              │  查 users 表获取用户
  │  {rooms, orders, users...}   │
  │←─────────────────────────────│
```

- Cookie 名：`ns_session`
- Cookie 属性：`HttpOnly`、`SameSite=Lax`、`Max-Age=86400`（24小时）
- Session 持久化：写入数据库/文件，重启不丢失
- 过期清理：启动时自动清理过期 session

## 功能模块详解

### 前台看板

前台看板是日常使用频率最高的页面，设计目标是"一眼看清所有房态"。

**桌面端布局：**
- 3 列自适应卡片网格，每个房型一个独立卡片
- 卡片数量自动适配（1-2 个房型用对应列数，3+ 个用 3 列）
- 每个卡片：标题栏（房型名 + 描述 + 统计 + 缩放按钮）+ 可滚动内容区
- 内容区按楼层分组，用标签 + 分割线分隔
- 房间格子紧凑（约 48px 高），显示房号 + 状态文字 + 状态角标
- 每个卡片独立滚动，一屏总览所有房型

**移动端布局：**
- 垂直滚动长页面
- 保留"按楼层 / 按房型"切换
- 房间格子同样紧凑

**房间卡片状态：**
| 状态 | 颜色 | 图标 | 说明 |
|------|------|------|------|
| 空闲 | 绿色 | 无 | 可入住/预订 |
| 预订 | 琥珀色 | 文件图标 | 已被预订 |
| 已住 | 红色 | 用户图标 | 有客人入住 |
| 维修/打扫 | 蓝色 | 工具图标 | 退房后待清洁 |

**缩放功能：**
- 每个房型卡片右上角有 − / 百分比 / + 控件
- 范围 60%~140%
- 调整单个房型的房间网格密度

### 房间操作流程

**散客入住（Walk-in）：**
1. 点击空闲房间卡片
2. 填写客人信息（姓名、电话、预计退房、押金）
3. 点击"办理入住"
4. 房间状态变为"已住"，生成订单

**预订入住：**
1. 点击空闲房间卡片
2. 填写预订信息
3. 点击"新建预订"
4. 房间状态变为"预订"
5. 客人到达后，点击预订房间
6. 补充/确认信息，点击"预订转入住"
7. 房间状态变为"已住"

**退房结算：**
1. 点击已住房间
2. 查看入住信息和押金
3. 点击"退房结算"
4. 房间状态变为"维修/打扫"

**打扫完成：**
1. 点击维修/打扫房间
2. 点击"打扫完成"
3. 房间状态恢复为"空闲"

### 后台管理

后台管理页面采用左侧导航栏 + 右侧内容区布局，包含 7 个功能模块：

**1. 后台首页**
- 今日开房/退房/预订数量统计卡片
- 订单概览（总数/进行中/已完成/已取消）
- 房态概览（各状态房间数量）
- 管理导航快捷入口

**2. 历史订单**
- 全部订单列表，含房间、状态、预订信息、开房信息、时间、押金、操作员
- 按状态筛选（全部/进行中/已完成/已取消）
- 按日期范围筛选（起始日期 + 截止日期）
- 关键词搜索（房号、房型、客人、电话、操作员）
- 分页（每页 8 条，首页/上一页/下一页/末页）
- 已完成/已取消订单支持删除（二次确认弹窗）
- 进行中订单禁止删除

**3. 账户管理**
- 账号列表（用户名、权限、创建时间、更新时间）
- 新增账号（用户名、权限、密码）
- 编辑账号（修改用户名、权限、密码）
- 删除账号（二次确认弹窗，不能删除当前登录账号，至少保留 1 个管理员）
- 分页（每页 10 条）
- 权限标识：管理员（蓝色徽章）/ 前台账号（灰色徽章）

**4. 系统设置**
- 分两个独立卡片，中间留空：
  - **退房默认时间**：偏移天数（0-30）、退房小时（0-23）、退房分钟（0-59）
  - **审计日志清理**：自动清理天数（1-365，禁止设为 0）
- 每个卡片独立保存按钮

**5. 房型配置**
- 房型列表（名称、描述、排序、房间数量）
- 新增/编辑/删除房型（二次确认弹窗）
- 分页（每页 10 条）
- 有房间关联的房型禁止删除

**6. 房间配置**
- 房间列表（房号、楼层、房型、状态）
- 单个新增房间
- 批量新增房间（起始房号-结束房号，自动分配楼层）
- 编辑/删除房间（二次确认弹窗）
- 分页（每页 15 条）
- 只有空闲房间可删除

**7. 审计日志**
- 全部操作记录（时间、操作员、房号、动作、详情）
- 关键词筛选（房号、动作、操作员）
- 显示总记录数
- 自动清理：根据系统设置的保留天数，每次请求时自动删除过期日志

### 安装向导（MySQL 版）

首次访问自动跳转安装向导，防止未配置就使用：

**第一步：数据库配置**
- 数据库地址、端口、用户名、密码、库名
- "测试连接"按钮：验证 MySQL 连接是否可用
- 连接成功后才能进入下一步

**第二步：管理员账号**
- 用户名、密码、确认密码
- 密码至少 6 位
- 不再使用默认的 admin/123456

**安装过程：**
- 自动创建 7 张数据库表
- 自动插入默认系统设置
- 自动创建管理员账号
- 生成配置文件 + 安装锁文件

**防重复安装：**
- `install.lock` 存在时，直接访问安装页面会自动跳转首页
- `install.lock` 存在时，`/api/install` 接口被拦截
- 需要重新安装时，删除 `install.lock` 和配置文件即可

### 实时同步

**Node.js 版（WebSocket）：**
- 手写 WebSocket 帧协议实现，无第三方依赖
- 服务端维护 socket 连接集合
- 任何数据变更后广播 `inventory_changed` 事件
- 客户端收到后自动刷新全部数据

**PHP 版（SSE）：**
- 数据变更时写入 `data/events.json`（含时间戳）
- 客户端通过 `EventSource("/api/events")` 监听
- 服务端每 2 秒检查事件文件，有新事件则推送
- 客户端收到后自动刷新全部数据

### 自定义确认弹窗

所有删除操作使用自定义卡片式确认弹窗，而非浏览器原生 `window.confirm()`：

- 居中显示，带遮罩层
- 红色标题栏 + 消息内容
- "取消" + "确认删除"按钮
- `animate-pop-in` 动画，与系统 UI 风格统一
- 点击遮罩层或取消按钮关闭

## 安全设计

### 密码安全

| 版本 | 算法 | 说明 |
|------|------|------|
| Node.js | scrypt + 随机 salt | Node.js 原生 crypto 模块 |
| PHP | bcrypt | `password_hash(PASSWORD_BCRYPT)` |

密码从不以明文存储，每次验证使用 `timingSafeEqual`（Node.js）或 `password_verify`（PHP）防止时序攻击。

### Session 安全

- Token 使用 `crypto.randomBytes(32)`（Node.js）或 `random_bytes(32)`（PHP）生成
- Cookie 设置 `HttpOnly`（防止 XSS 读取）
- Cookie 设置 `SameSite=Lax`（防止 CSRF）
- Session 24 小时自动过期
- 登出时服务端立即删除 session 记录

### 权限控制

- 两级权限：`ADMIN`（管理员）和 `STAFF`（前台账号）
- STAFF 只能操作房间（入住/退房/预订），不能访问后台管理
- ADMIN 可以访问全部功能
- 至少保留 1 个管理员账号，防止锁死
- 不能删除当前登录的账号

### 安装安全

- 安装完成后生成锁文件，防止重复安装
- 安装锁存在时，所有业务 API 返回 403
- 配置文件存储数据库凭据，部署时应设置适当文件权限

## 性能考虑

### JSON 文件版本

- 每次 API 请求全量读取 `db.json`（同步 `readFileSync`）
- 每次数据变更全量写入 `db.json`（同步 `writeFileSync`）
- 80 间房 + 几十条订单时性能可接受（文件 < 100KB）
- 不适合高并发场景

### MySQL 版本

- 使用连接池（Node.js: mysql2 pool, PHP: PDO persistent connection）
- 事务保证数据一致性（`FOR UPDATE` 行锁）
- 索引优化：`room_number`、`status`、`room_id`、`created_at` 等字段建索引
- Inventory 接口使用 LEFT JOIN 一次性查询，减少数据库往返
- 审计日志启动时自动清理过期数据

## 设计决策记录

### 为什么不用 Express / Koa？

项目只有 ~15 个 API 端点，手写路由匹配的代码量与 Express 的 router 配置差不多，但省去了整个框架依赖。对于这个规模的项目，框架带来的便利不足以抵消其复杂度。

### 为什么前端用 CDN 而不是 npm 构建？

目标用户是小旅馆老板，他们不会配置 Node.js 构建环境。CDN 引入 React + Tailwind 意味着只需要一个 HTML 文件就能运行整个前端，部署时直接上传即可。

### 为什么 WebSocket 手写而不使用 ws 库？

ws 库需要 npm 安装，增加了部署复杂度。WebSocket 的帧协议并不复杂（服务端只需构造 0x81 开头的帧），手写 ~20 行代码就能实现推送功能。

### 为什么 PHP 版用 SSE 而不是 WebSocket？

PHP 的运行模型是请求-响应，不适合维护长连接。SSE（Server-Sent Events）是 HTTP 单向推送，PHP 可以通过 `flush()` 实现，且不需要特殊的服务端配置。功能上等价于 WebSocket 的服务端推送子集。

### 为什么密码哈希算法不同？

Node.js 原生 `crypto` 模块支持 scrypt，PHP 原生 `password_hash` 使用 bcrypt。两者都是安全的密码哈希算法，选择各自语言的原生方案避免了额外依赖。由于三个版本使用独立数据库，不存在哈希兼容性问题。

### 为什么审计日志要自动清理？

小旅馆每天几十条操作记录，一年就是上万条。日志表没有业务查询价值（主要用于追溯），设置自动清理（默认 3 天）防止数据库无限膨胀。管理员可在系统设置中调整保留天数（1-365 天），但不能设为 0（禁止关闭清理）。
